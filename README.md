# 车联网单车行程与事件提取（Trip & Event）逻辑说明文档

## 1. 概述
本文档详细规定了车联网（Telematics）系统中，基于高频车辆上报明细流（DWD层）提取**一级主行程（Level 1 Trip）**与**二级明细事件（Level 2 Sub-Events）**的底层状态机流转逻辑。

本方案采用“批流一体”架构设计，一套代码同时支持实时 Kafka 流计算与离线 Paimon T+1 跑批。底层依靠 Spark Structured Streaming 的 `applyInPandasWithState` 算子实现状态增量追踪，在保证极低内存开销的同时，完整捕获各类事件的**起止时间**与**双端 GPS 经纬度**。

---

## 2. 核心概念定义

### 2.1 状态机内部相位 (Phase)
车辆在流处理框架内存中的生命周期由以下三个核心相位控制：
* **OFF (离线/休眠)**：初始状态或确认行程结束后的状态。
* **ACTIVE (活跃/行驶中)**：车辆上电或有车速，正在持续收集行程轨迹与检测二级事件。
* **HANG_OFF (挂起/观察期)**：接收到下电信号（`syspowermod=0`），但由于网络抖动或临时熄火，不立即切断行程，进入最长 **15分钟** 的观察期。

### 2.2 输出数据类型 (`row_type`)
为将多粒度数据沉淀至同一张数仓表中，输出数据分为两类：
* **`TRIP` (一级行程)**：宏观层面的单次驾驶或驻车过程。包含 `DRIVING`（只要行程内有车速即为驾驶）和 `PARKING`（全程无车速为驻车）。
* **`EVENT` (二级事件)**：在一级行程内部发生的各类动作或持续状态，如“蠕行”、“打开后备箱”等。

---

## 3. 一级主行程（Level 1）切分逻辑

一级主行程的目的是将连续不断的车辆报文切割为具备业务意义的“一段一段”的行程。

* **启动条件**：当车辆处于 `OFF` 状态时，一旦接收到上电信号（`syspowermod == 2`）或车速大于 0 (`speed > 0`)，立即开启一段新行程，记录 `trip_start`。
* **挂起条件**：当车辆处于 `ACTIVE` 状态时，接收到下电信号（`syspowermod == 0`），行程不立即结束，而是进入 `HANG_OFF` 状态。
* **结算（闭合）条件**：
    1.  **正常超时切断**：处于 `HANG_OFF` 状态的车辆，如果连续 **15分钟（900秒）** 没有重新上电或产生车速，则判定上一段行程真实结束。行程结束时间回溯至下电时刻 `hang_start`。
    2.  **异常断层切断**：在 `ACTIVE` 或 `HANG_OFF` 状态下，如果当前报文时间与上一条报文时间发生跳跃式中断（时间差 > **15分钟**），为防止脏数据污染，强制切断并结算上一行程，结束时间记为上一条报文的时间。
    3.  **水位线超时（Watermark Timeout）**：车辆长时间断网离线，Spark 状态机触发 EventTime 超时（15分钟），强制结算残留行程。

---

## 4. 二级事件（Level 2）判定与提取逻辑

二级事件仅在车辆处于 **ACTIVE（活跃状态）** 时进行检测。采用了**字典增量追踪（Dictionary Incremental Tracking）**模型，当条件首次满足时捕获 `start_time`, `start_lon`, `start_lat` 并入内存字典；当条件不再满足时出栈，并与当前时间戳比对时长，达到阈值则生成事件。

### 4.1 二级事件明细规则表 (共23项)

| 事件分类 | 事件名称 (中文) | 事件代码 (英文) | 触发条件 / 判定逻辑 | 持续时长要求 |
| :--- | :--- | :--- | :--- | :--- |
| **首尾边界** | 驾乘准备事件 | `PARK_BEFORE_RUN_EVENT` | 从行程开始（上电）到第一条 `车速 > 0` 的过程。 | 无限制 |
| | 驻留事件 | `PARK_AFTER_RUN_EVENT` | 从行程内最后一次 `车速 > 0` 到行程结束（下电）的过程。 | 无限制 |
| **车速区间** | 蠕行事件 | `CREEP_RUN_EVENT` | `0 < 车速 <= 10 km/h` 的连续过程。 | **≥ 5 分钟** |
| | 拥堵事件 | `TRAFFIC_JAM_EVENT` | `10 < 车速 <= 20 km/h` 的连续过程。 | **≥ 5 分钟** |
| | 低速行驶事件 | `LOW_SPEED_EVENT` | `0 < 车速 <= 40 km/h` 的连续过程。 | 只要发生即记录 |

| | 超速行驶事件 | `OVER_SPEED_EVENT` | `车速 > 120 km/h` 的连续过程。 | 只要发生即记录 |
| | 驻车事件 | `PARK_RUNNING_EVENT` | `车速 == 0` 的连续过程（在ACTIVE行程中）。 | 只要发生即记录 |
| **硬件开关** | 车机上下电事件 | `ON_OFF_IVI_EVENT` | 车机电源状态持续为 ON 的过程。 | 只要发生即记录 |

> **注**：二级事件的起止判定是完全相互独立的，可以同时重叠存在（例如车辆完全可以同时处于“蠕行事件”、“低速行驶事件”与“主驾座椅加热事件”中）。

### 4.2 兜底结算规则 (Flush)
如果一级主行程发生结算（因超时或断层被切断），当前所有记录在内存字典中尚未闭合的二级事件，将**强制执行闭合结算**。以强制切断时间作为事件结束时间，并校验持续时长是否满足阈值，满足则下发。

## 5. 输出数据规范 (Output Schema)

最终落盘至 ClickHouse/ByteHouse ADS 层表的字段定义如下：

| 字段名称 | 数据类型 | 说明 / 样例 |
| :--- | :--- | :--- |
| `vin` | String | 车辆唯一标识码 |
| `row_type` | String | 数据粒度标示：`TRIP` (一级行程), `EVENT` (二级事件) |
| `trip_type` | String | 若是TRIP，值为 `DRIVING`或`PARKING`；若是EVENT，值为事件英文名 |
| `start_time` | Timestamp | 行程或事件开始的精确时间 |
| `end_time` | Timestamp | 行程或事件结束的精确时间 |

| `avg_speed` | Double | 仅 TRIP 记录计算：行程平均时速（EVENT统一为0） |

| `start_lat` | Double | 仅 EVENT 记录：事件触发时刻车辆所在的 GPS 纬度 |
| `end_lon` | Double | 仅 EVENT 记录：事件结束时刻车辆所在的 GPS 经度 |
| `end_lat` | Double | 仅 EVENT 记录：事件结束时刻车辆所在的 GPS 纬度 |
