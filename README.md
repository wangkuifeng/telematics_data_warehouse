# 车辆行程切分状态机引擎设计文档

## 一、核心架构图：状态机流转逻辑 (State Machine)

                  [ 初始状态：OFF ]
                           │
   (触发上电 syspowermod==2 或 车速 vehspd>0)
                           │
                           ▼
                 ┏━━━━━━━━━━━━━━━━━━┓
                 ┃      ACTIVE      ┃ ◄━━━━━┓
                 ┃   (行程进行中)   ┃       ┃
                 ┗━━━━━━━━━━━━━━━━━━┛       ┃ (15分钟内重新上电
                           │                ┃  或产生车速：
                           │                ┃  行程合并连着算)
                   (接收到下电信号           ┃
                    syspowermod==0)         ┃
                           │                ┃
                           ▼                │
                 ┏━━━━━━━━━━━━━━━━━━┓       │
                 ┃     HANG_OFF     ┃━━━━━━━┛
                 ┃ (15分钟缓冲观察期)┃
                 ┗━━━━━━━━━━━━━━━━━━┛
                           │
               (熬满 15 分钟未收到上电
                或 全局 Watermark 超时)
                           │
                           ▼
                 【确认结束，闭合结算】
                           │
               (分析整段行程的速度序列)
                           │
     ├─ 包含任意时刻速度 > 0 ──► 产出【行驶行程 (DRIVING)】
     └─ 全程所有点位速度 == 0 ──► 产出【驻车行程 (PARKING)】




### 2.3 核心业务规则矩阵

| 规则维度 | 判定条件 | 引擎系统动作 | 业务设计意图 |
| :--- | :--- | :--- | :--- |
| **规则 1：行程开始** | 收到 `syspowermod == 2`（上电）或连续收到 `vehspd_2011002 > 0` 的数据。 | 立即初始化状态，标记 `trip_start = ts`，进入 `ACTIVE` 阶段，开启速度序列收集。 | 同时支持硬信号（上电）与软信号（车速补偿），防止部分 T-Box 启动上电信号丢失造成漏计。 |
| **规则 2：长时连续性** | 车辆处于 `ACTIVE` 状态，无论速度是否为 0，只要未下电。 | **持续保持 ACTIVE 状态**，将速度连续追加至 `trip_speeds`。**彻底移除长时怠速截断规则**。 | 确保路边长时间未熄火等人、严重拥堵等行为被完整保留在单一的大行程内，保证行程宏观连续性。 |
| **规则 3：下电观察期** | 收到 `syspowermod == 0`（下电信号）。 | 状态转入 `HANG_OFF`，记录 `hang_start = ts`。在墙上挂起一个 **15 分钟** 的事件时间闹钟。 | 防抖设计。避免进加油站、便利店短暂买水等正常停靠行为将一段完整拉练切成多段碎数据。 |
| **规则 4：断网兜底结算** | 车辆驶入盲区（地库/山区）导致心跳骤断，或观察期满 15 分钟未复位。 | 全局 Watermark 越过设定的 Timeout 闹钟时，强制关闭该车通道，**结束时间回溯至最后一条有效日志或挂起点**，闭合行程。 | 防止死锁。避免无下电信号导致的行程无限悬挂，以及 RocksDB 底层状态无限膨胀导致的 OOM 风险。 |
